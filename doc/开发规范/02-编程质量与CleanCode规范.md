# 编程质量与 Clean Code 规范

本规范旨在确保代码的一致性、可维护性和高质量。在编写或审查代码时，请遵循以下准则。

## 编程质量准则

### 验证信息
在呈现或实现信息之前，务必进行核实。不要在没有明确证据的情况下做出假设或推测。

### 逐步变更
代码变更应逐个文件进行，以便于及时发现错误和进行代码审查。

### 保持简洁
避免在注释或文档中提供冗余的理解反馈。

### 保护现有代码
不要删除无关的代码或功能。在修改时务必注意保留现有的项目结构和逻辑稳定性。

### 真实链接
在引用文件时，务必提供文件的真实路径或链接，而不是占位符。

## Clean Code 准则

### 常量优于魔术字
- 使用命名常量替换硬编码的值。
- 使用能够解释该值用途的描述性常量名称。
- 将常量置于文件顶部或专门的常量定义文件中。

### 有意义的命名
- 变量、函数和类的命名应揭示其用途。
- 命名应解释其存在的原因以及使用方式。
- 除非是业界通用的缩写，否则避免使用缩写。

### 聪明的注释
- 不要注释代码“做了什么”——应该让代码实现自文档化。
- 使用注释来解释“为什么”要以某种方式实现。
- 为 API、复杂算法和非显而易见的副作用编写文档。

### 单一职责 (SRP)
- 每个函数应该只做一件事。
- 函数应保持短小且专注。
- 如果一个函数需要注释来解释它在做什么，那么它可能需要被拆分。

### DRY 原则 (Don't Repeat Yourself)
- 将重复的代码提取为可复用的函数。
- 通过适当的抽象共享公共逻辑。
- 保持单一可信数据源（Single Source of Truth）。

### 整洁的结构
- 将相关的代码组织在一起。
- 按逻辑层级组织代码。
- 使用一致的文件和文件夹命名约定。

### 封装
- 隐藏实现细节，暴露简洁的接口。
- 将嵌套的条件判断提取到命名良好的函数中。

### 持续维护
- 持续进行重构，尽早修复技术债。
- 离开时代码应比你发现它时更整洁。

### 测试规范
- 在修复 Bug 之前编写测试。
- 保持测试的可读性和可维护性。
- 完善边界情况和错误条件的测试覆盖。
