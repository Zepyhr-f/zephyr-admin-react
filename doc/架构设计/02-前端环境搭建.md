# Phase 0 · 前端环境搭建

> **目标**：让每个开发者都能在本机以完全一致的方式运行 Zephyr Admin 前端项目，从拉代码到看到页面不超过 10 分钟。

---

## 2. 版本锁定策略

> **原则**：工具版本不统一是团队协作中最常见的"幽灵 Bug"来源，必须强制锁定。

### 2.1 Node 版本锁定

项目根目录的 `.nvmrc` 文件中写入 `22`，所有人进入项目目录后执行 `nvm use` 即可自动切换到正确版本。

`package.json` 中的 `engines` 字段同步声明：
```json
{
  "engines": {
    "node": "22.*"
  }
}
```

### 2.2 pnpm 版本锁定

`package.json` 中的 `packageManager` 字段声明具体版本，pnpm 会在不匹配时给出警告：
```json
{
  "packageManager": "pnpm@10.8.0"
}
```

## 3. 后端依赖服务启动（Docker Compose）

前端本地开发需要以下后端基础服务运行。通过 OrbStack + Docker Compose 一键启动，无需手动配置。

### 3.1 需要的服务清单

| 服务 | 端口 | 说明 |
|---|---|---|
| **MySQL 8.0** | `3306` | 主数据库 |
| **Redis 7.2** | `6379` | 缓存与 Token 存储 |
| **Nacos** | `8848` | 注册中心与配置中心 |
| **MinIO** | `9000 / 9001` | 本地对象存储（文件上传） |

### 3.2 启动方式

项目提供了 `z-web.sh` 脚本，统一管理前端开发相关命令。

```bash
# 启动后端依赖服务（MySQL + Redis + Nacos + MinIO）
./z-web.sh infra:up

# 确认各服务健康状态
./z-web.sh infra:status
```

> **注意**：如果只做前端 UI 开发，可以不启动后端服务，全程使用 MSW Mock 数据。

## 4. 前端项目初始化 (从零搭建)

### 4.1 创建基础脚手架

我们使用 Vite 6 + React 19 作为项目基础。

```bash
# 1. 使用 pnpm 创建项目
pnpm create vite zephyr-web --template react-ts

# 2. 进入目录并设置 Node 版本
cd zephyr-web
echo "22" > .nvmrc
nvm use

# 3. 安装基础依赖
pnpm install
```

### 4.2 安装核心技术栈依赖

按模块安装我们选定的现代化技术栈：

```bash
# 1. UI 框架与样式工具 (Tailwind v4 + Ant Design 5)
pnpm add antd @ant-design/cssinjs lucide-react clsx tailwind-merge
pnpm add -D tailwindcss @tailwindcss/vite

# 2. 状态管理与数据流
pnpm add zustand @tanstack/react-query axios

# 3. 路由、动画与表单
pnpm add react-router motion react-hook-form zod

# 4. 开发工具与规范 (Biome + Lefthook)
pnpm add -D @biomejs/biome lefthook @types/node
```

### 4.3 初始化规范配置

```bash
# 初始化 Biome (替代 ESLint/Prettier)
pnpm biome init

# 初始化 Lefthook (Git Hooks 管理)
pnpm lefthook install
```

### 4.2 配置本地环境变量

项目使用 `.env` 文件族管理环境变量，规则如下：

| 文件 | 作用 | 是否提交 Git |
|---|---|---|
| `.env` | 通用默认值（所有环境共享） | ✅ 提交 |
| `.env.development` | 开发环境专用变量 | ✅ 提交 |
| `.env.production` | 生产环境专用变量 | ✅ 提交 |
| `.env.local` | 本地个人覆盖（优先级最高） | ❌ 不提交，`.gitignore` 忽略 |

**必须配置的变量**：

```bash
# .env.local（本地个人配置，不提交）

# 后端 API 地址（开发时指向本地 Gateway 服务）
VITE_API_BASE_URL=http://localhost:9000

# 是否启用 MSW Mock（true = 纯前端 Mock，false = 请求真实后端）
VITE_ENABLE_MOCK=true
```

### 4.3 启动开发服务器

```bash
pnpm dev
```

启动成功后，终端显示本地地址（通常是 `http://localhost:5173`）。

## 5. MSW Mock 数据说明

> **设计理念**：前端开发不应该被后端接口进度阻塞。MSW（Mock Service Worker）在浏览器层面拦截请求，让前端可以独立开发，联调时只需关闭 Mock 开关。

### 5.1 MSW 工作原理

```
浏览器发起 API 请求
    ↓
Service Worker 拦截（浏览器层，不经过网络）
    ↓
MSW Handler 匹配请求路径
    ↓
返回 Mock 数据（完全模拟真实响应格式）
```

### 5.2 Mock 的开关方式

- **开启 Mock**：`.env.local` 中设置 `VITE_ENABLE_MOCK=true`
- **关闭 Mock**：设置 `VITE_ENABLE_MOCK=false`，所有请求发往真实后端

## 6. Git Hooks 验证

项目使用 `Lefthook` 管理 Git Hooks。`pnpm install` 时会自动安装（通过 `preinstall` 脚本执行 `lefthook install`）。

### 6.1 已配置的 Hook

| Hook | 触发时机 | 执行内容 |
|---|---|---|
| `pre-commit` | `git commit` 前 | Biome lint 检查，失败则阻止提交 |
| `commit-msg` | 输入提交信息后 | Commitlint 格式检查 |

### 6.2 Commit 信息规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <subject>

type 可选值：
  feat     - 新功能
  fix      - Bug 修复
  docs     - 文档变更
  style    - 代码格式（不影响逻辑）
  refactor - 重构
  perf     - 性能优化
  test     - 测试相关
  chore    - 构建/工具链变更
```

示例：
```
feat(user): 新增用户列表分页查询
fix(auth): 修复 Token 过期后未跳转登录页的问题
```

- [ ] `pnpm dev` 启动无报错
- [ ] 浏览器访问 `http://localhost:5173` 能看到登录页
