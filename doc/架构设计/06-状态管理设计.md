# 前端状态管理设计

> 本文档阐述在 Zephyr Admin 前端如何划分状态域、使用 Zustand 与 TanStack Query 协同工作、持久化策略以及最佳实践，确保数据流清晰、维护成本低。

---

## 1. 状态划分原则

| 类别 | 管理工具 | 适用场景 | 示例 |
|---|---|---|---|
| **服务端状态** | TanStack Query | 需要缓存、自动刷新、分页、请求合并的后端数据 | 用户列表、角色列表、审计日志等
| **客户端 UI 状态** | Zustand | 页面交互、全局 UI 配置、临时本地状态 | 侧边栏折叠、主题切换、多标签页、表单弹窗开关 |
| **持久化本地状态** | Zustand + persist 中间件 | 需要在刷新后保持的状态 | 语言偏好、暗黑模式、最近访问页面 |

> **原则**：所有直接来自后端的业务数据必须走 Query；纯 UI 状态走 Store；二者不相互混用。

## 2. Zustand Store 组织结构

```
store/
├── use-auth-store.ts          # 登录 token、用户信息、权限列表
├── use-user-store.ts          # 当前用户详情（从 Query 缓存中抽取）
├── use-theme-store.ts         # 主题（亮/暗）+ 颜色方案
├── use-layout-store.ts        # 侧边栏收缩、页面宽度、面包屑展开状态
├── use-tab-store.ts           # 多标签页（打开、激活、关闭）
├── use-settings-store.ts      # 全局设置（语言、时间格式）
└── index.ts                   # 汇总导出
```

### 2.1 `use-auth-store.ts`
```tsx
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

export interface AuthState {
  token: string | null;
  refreshToken: string | null;
  userInfo: UserInfo | null;
  permissions: string[];
  setToken: (t: string) => void;
  setUserInfo: (info: UserInfo) => void;
  setPermissions: (perms: string[]) => void;
  logout: () => void;
}

export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      token: null,
      refreshToken: null,
      userInfo: null,
      permissions: [],
      setToken: (t) => set({ token: t }),
      setUserInfo: (info) => set({ userInfo: info }),
      setPermissions: (perms) => set({ permissions: perms }),
      logout: () => set({ token: null, refreshToken: null, userInfo: null, permissions: [] }),
    }),
    { name: 'auth-store', getStorage: () => localStorage }
  )
);
```

- **持久化**：使用 `localStorage`，保证刷新后仍保持登录状态（前提后端 token 仍有效）。
- **安全提醒**：生产环境建议使用 `sessionStorage` 或加密存储。

## 3. TanStack Query 使用规范

| 场景 | Query Key 规范 | 示例 | 备注 |
|---|---|---|---|
| 列表查询 | `['entity', { page, size, filters }]` | `['user', { page: 1, size: 20, status: 'active' }]` | 防止同实体不同过滤条件冲突 |
| 详情查询 | `['entity', id]` | `['user', 123]` | 自动缓存单条数据 |
| 变更（POST/PUT/DELETE） | `mutationKey: ['entity', 'create']` 等 | `['user', 'delete']` | 成功后手动 `invalidateQueries(['user'])` 触发列表刷新 |

### 3.1 基础封装 `src/api/client.ts`
```ts
import axios from 'axios';
import { useAuthStore } from '@/store/use-auth-store';

const instance = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 12_000,
});

// 请求拦截器：注入 token
instance.interceptors.request.use((config) => {
  const token = useAuthStore.getState().token;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// 响应拦截器：统一错误处理
instance.interceptors.response.use(
  (res) => res,
  (err) => {
    if (err.response?.status === 401) {
      // token 失效统一处理
      useAuthStore.getState().logout();
    }
    return Promise.reject(err);
  }
);

export default instance;
```

### 3.2 Query Hook 示例
```tsx
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import client from '@api/client';

export const useUserList = (params) => {
  return useQuery(['user', params], () =>
    client.get('/api/system/user', { params }).then((res) => res.data)
  );
};

export const useCreateUser = () => {
  const qc = useQueryClient();
  return useMutation((payload) => client.post('/api/system/user', payload), {
    onSuccess: () => qc.invalidateQueries(['user']),
  });
};
```

## 4. 状态持久化方案

- **全局 UI 设置**（暗黑模式、语言）使用 `persist` 中间件，存储在 `localStorage`。
- **敏感信息**（Token）默认使用 `sessionStorage`，仅在浏览器会话期间有效，防止跨标签持久化泄漏。
- **手动同步**：在 `useEffect` 中监听 Query 缓存变化，将必要数据写入 Store（如 `useUserStore` 从 `useUserList` 中提取当前用户信息）。

## 5. 开发与调试指南

1. **开启 React Query DevTools**：在 `src/main.tsx` 中加入 `<ReactQueryDevtools initialIsOpen={false} />`，便于实时查看缓存状态。
2. **Zustand 调试**：使用 Chrome Redux DevTools 插件的 `Zustand` 支持，或在 Store 中加入 `devtools` 中间件。
3. **类型安全**：所有 Query 返回值使用 `R<T>` 统一响应体包装，配合 `api-types.ts` 生成对应 TS 接口。

## 6. 常见坑与最佳实践

| 症状 | 可能原因 | 解决方案 |
|---|---|---|
| 页面切换后列表空白 | Query key 未包含分页/过滤参数 | 确保 `useUserList({ page, size })` 的 key 完全对应参数 |
| 多次请求同一数据 | 未使用 `staleTime` 或 `cacheTime` | 为不频繁变更的数据设置 `staleTime: 5 * 60 * 1000` |
| Store 状态被意外覆盖 | 直接修改 Store 对象属性 | 始终使用 `set` 回调方式更新状态 |
| Token 失效后仍继续请求 | 拦截器未及时清除 token | 在响应拦截器中统一 `logout` 并跳转登录页 |

## 7. 迁移与演进路线图

- **V1**：Zustand + Query 基础实现（已完成）
- **V2**：引入 `react-keep-alive` 对高频页面做缓存，结合 `keepAlive` meta
- **V3**：统一 `useStore` 与 `useQuery` 返回的状态结构，实现 `useSelect` 直接读取缓存而不触发网络请求
- **V4**：将状态持久化迁移到 IndexedDB（使用 `idb-keyval`）提升大数据持久化能力
