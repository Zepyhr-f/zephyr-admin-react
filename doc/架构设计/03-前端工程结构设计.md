# Phase 1 · 前端工程结构设计

> **目标**：建立清晰、可扩展的目录分层，让每个文件都有明确的归属，让每个新成员都能快速定位代码位置。

---

## 1. 设计原则

| 原则 | 说明 |
|---|---|
| **就近原则** | 与某个页面强相关的组件、Hook、类型，放在该页面目录下，而非全局目录 |
| **单一职责** | 每个目录只承担一种职责，不混用（如 `api/` 只放接口请求，不放业务逻辑） |
| **清晰分层** | UI 原子组件 → 业务组件 → 页面组件，三层严格隔离，下层不依赖上层 |
| **按域聚合** | 同一业务域的文件（如用户管理的列表、表单、详情）放在同一目录下，而非按文件类型分散 |

```
zephyr-web/

├── biome.json              # Biome lint 配置
├── tailwind.config.ts      # TailwindCSS 配置
├── lefthook.yml            # Git Hooks 配置
└── package.json
```

## 3. `src/` 目录详细设计

```
src/
├── main.tsx                # 应用程序入口，挂载 React App
├── App.tsx                 # 根组件，配置 Provider 树
├── global.css              # 全局样式（CSS 变量、Reset、基础排版）
├── vite-env.d.ts           # Vite 环境变量类型声明
├── global-config.ts        # 全局静态配置（不随 env 变化的常量）
│
├── api/                    # 【接口层】所有后端接口请求定义
├── assets/                 # 【静态资源】图片、SVG、字体等
├── components/             # 【全局业务组件】跨页面复用的业务组件
├── hooks/                  # 【全局自定义 Hook】跨页面复用的 Hook
├── layouts/                # 【布局组件】各种页面布局骨架
├── locales/                # 【国际化】语言包文件
├── pages/                  # 【页面层】所有路由对应的页面
├── routes/                 # 【路由配置】路由定义与守卫
├── store/                  # 【状态管理】Zustand Store
├── theme/                  # 【主题配置】Ant Design Token、TailwindCSS 变量
├── types/                  # 【全局类型】共享 TypeScript 类型定义
├── ui/                     # 【UI 原子组件】无业务逻辑的纯展示组件
├── utils/                  # 【工具函数】纯函数工具
└── _mock/                  # 【Mock 数据】MSW Handler 定义
```

## 4. 各目录职责详解

### 4.1 `api/` — 接口层

**职责**：定义所有与后端的数据通信，是前后端的唯一边界。

**结构**：
```
api/
├── client.ts           # Axios 实例配置（拦截器等）
├── index.ts            # 统一导出入口
├── auth/               # 认证相关接口（登录、登出、获取用户信息）
├── system/             # 系统管理接口
│   ├── user.ts         # 用户管理
│   ├── role.ts         # 角色管理
│   ├── menu.ts         # 菜单管理
│   ├── dept.ts         # 部门管理
│   └── dict.ts         # 字典管理
└── biz/                # 业务模块接口（按业务域继续拆分）
```

**规范**：
- 每个文件对应一个业务实体
- 只做"发请求 + 返回原始数据"，不包含任何业务逻辑
- 函数命名格式：`动词 + 实体 + 可选描述`，如 `getUserList`、`createUser`、`deleteUserById`

---

### 4.2 `components/` — 全局业务组件

**职责**：跨多个页面复用的业务组件，包含具体业务语义。

**结构**：
```
components/
├── permission/         # 权限控制组件（按钮级权限显隐）
├── upload/             # 文件上传组件（基于 MinIO）
├── icon-selector/      # 图标选择器
├── dict-tag/           # 字典标签展示组件
├── user-avatar/        # 用户头像组件
└── ...
```

**与 `ui/` 的区别**：
- `ui/` 是无业务逻辑的纯 UI 组件（Button、Modal、Badge 等样式封装）
- `components/` 是有业务含义的组件，可能调用 API 或读取 Store

---

### 4.3 `layouts/` — 布局组件

**职责**：定义页面的骨架结构，是路由与具体页面之间的"外壳"。

**结构**：
```
layouts/
├── admin-layout/       # 管理后台主布局（侧边栏 + 顶栏 + 内容区 + 多标签页）
│   ├── index.tsx
│   ├── sidebar/        # 侧边导航栏
│   ├── header/         # 顶部工具栏
│   ├── tab-bar/        # 多标签页 Tab 栏
│   └── content/        # 主内容区
├── auth-layout/        # 认证布局（登录、注册、忘记密码页面）
└── blank-layout/       # 空白布局（全屏展示、打印页面等）
```

---

### 4.4 `pages/` — 页面层

**职责**：路由对应的页面组件，每个目录对应一个路由或路由分组。

**结构**：
```
pages/
├── login/              # 登录页
├── dashboard/          # 首页 Dashboard
├── system/             # 系统管理模块
│   ├── user/           # 用户管理
│   │   ├── index.tsx   # 用户列表页
│   │   ├── components/ # 页面私有组件（用户表单弹窗、角色选择器等）
│   │   ├── hooks/      # 页面私有 Hook
│   │   └── types.ts    # 页面私有类型定义
│   ├── role/
│   ├── menu/
│   ├── dept/
│   └── dict/
├── exception/          # 异常页面（403、404、500）
└── ...（按业务域继续扩展）
```

**关键设计：就近原则**
- 每个页面目录下可有自己的 `components/`、`hooks/`、`types.ts`
- 这些私有文件不对外暴露，只服务于该页面
- 只有当组件/Hook 被 2 个以上页面使用时，才提升到全局 `components/` 或 `hooks/`

---

### 4.5 `routes/` — 路由配置

**职责**：集中管理所有路由定义，包含路由守卫逻辑。

**结构**：
```
routes/
├── index.tsx           # 路由根配置，组合所有路由模块
├── auth-guard.tsx      # 路由守卫（登录检查、权限检查）
├── public-routes.ts    # 公开路由（登录页、错误页）
├── admin-routes.ts     # 后台路由（需要登录，动态注入）
└── route-meta.ts       # 路由 Meta 类型定义
```

---

### 4.6 `store/` — 状态管理

**职责**：Zustand 全局状态 Store，按业务域拆分文件。

**结构**：
```
store/
├── use-user-store.ts       # 用户信息（当前登录用户、权限列表）
├── use-auth-store.ts       # 认证状态（Token、登录状态）
├── use-theme-store.ts      # 主题配置（暗黑模式、颜色方案）
├── use-settings-store.ts   # 应用设置（语言、侧边栏收缩等）
└── use-tab-store.ts        # 多标签页状态
```

---

### 4.7 `theme/` — 主题配置

**职责**：统一管理所有视觉主题配置，是设计系统的代码实现层。

**结构**：
```
theme/
├── antd-token.ts       # Ant Design Design Token 覆盖配置
├── css-variables.css   # 全局 CSS 自定义变量（供 TailwindCSS 使用）
├── dark-token.ts       # 暗黑模式 Token
└── color-palette.ts    # 颜色调色板定义
```

---

### 4.8 `types/` — 全局类型定义

**职责**：跨模块复用的 TypeScript 类型，特别是与后端接口对应的数据模型。

**结构**：
```
types/
├── api.d.ts            # API 响应通用类型（统一响应体 R<T> 等）
├── router.d.ts         # 路由 Meta 类型扩展
├── global.d.ts         # 全局声明（window 扩展、env 变量类型）
└── entity/             # 业务实体类型
    ├── user.d.ts
    ├── role.d.ts
    └── menu.d.ts
```

## 5. 文件命名规范

| 文件类型 | 命名规范 | 示例 |
|---|---|---|
| **React 组件文件** | PascalCase | `UserTable.tsx`, `RoleFormModal.tsx` |
| **页面目录/入口** | kebab-case | `user-management/`, `index.tsx` |
| **Hook 文件** | use- 前缀 + camelCase | `use-user-list.ts`, `use-permission.ts` |
| **工具函数文件** | kebab-case | `format-date.ts`, `validate-phone.ts` |
| **Store 文件** | use- 前缀 + kebab-case | `use-user-store.ts` |
| **API 文件** | kebab-case | `user-api.ts`, `role-api.ts` |
| **类型文件** | kebab-case + `.d.ts` 或 `.ts` | `user.d.ts`, `api-types.ts` |
| **样式文件** | 与组件同名 | `UserTable.module.css` |

## 6. 路径别名配置

为避免深层相对路径（如 `../../../../components`），配置 Vite 路径别名：

| 别名 | 实际路径 | 使用示例 |
|---|---|---|
| `@/` | `src/` | `import { useUserStore } from '@/store/use-user-store'` |
| `@api/` | `src/api/` | `import { getUserList } from '@api/system/user'` |
| `@comp/` | `src/components/` | `import DictTag from '@comp/dict-tag'` |
| `@pages/` | `src/pages/` | `import Login from '@pages/login'` |

---

## 7. 导入顺序规范

每个文件的 `import` 语句按以下顺序排列（由 Biome 自动格式化强制执行）：

```
1. React 核心（react, react-dom）
2. 第三方库（ant-design, zustand, react-router, axios...）
3. 项目内部 - 全局模块（@/store, @/api, @/utils...）
4. 项目内部 - 相对路径（./components, ../hooks...）
5. 类型导入（import type ...）
6. 样式导入（import './xxx.css'）
```
