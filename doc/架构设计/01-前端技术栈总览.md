## 1. 技术选型一览（按层次）

| 层次 | 技术/库 | 说明（为什么用它） |
|---|---|---|
| 运行环境 | Node.js + pnpm | 统一工具链、可复现安装；pnpm 适合 monorepo/workspace |
| 核心框架 | React | 组件化与生态成熟，适合中大型管理后台 |
| 构建工具 | Vite | HMR 快、配置轻、生态成熟 |
| 语言 | TypeScript（strict） | 全量类型安全，减少联调与重构成本 |
| UI 组件 | Ant Design | 企业后台通用组件齐全，适合快速交付 |
| 样式体系 | TailwindCSS +（可选）CSS-in-TS | 原子化效率高；复杂主题/Token 可用 CSS 变量或 CSS-in-TS 收敛 |
| 路由 | React Router（数据路由能力可选） | 路由分层清晰，守卫/懒加载实现成本低 |
| 数据请求 | Axios | 拦截器统一注入 Token、错误处理与重试策略 |
| 服务端状态 | TanStack Query | 缓存、并发去重、分页与后台刷新开箱即用 |
| 客户端状态 | Zustand | 轻量、低样板代码，适合 UI 状态与少量全局状态 |
| 表单校验 | React Hook Form + Zod | 性能好、校验规则可复用（schema 驱动） |
| Mock | MSW | 浏览器层拦截，前端可不依赖后端即可开发 |
| 代码质量 | Biome | 格式化 + lint 一体化，减少工具链分裂 |
| Git 规范 | Lefthook + Commitlint | pre-commit / commit-msg 阶段强约束，保障质量基线 |

## 2. 关键边界：状态分工（必须遵守）

> **原则**：来自后端的数据走 Query；纯 UI/交互状态走 Store；避免“一个数据两处真相”。

- **TanStack Query**：列表、详情、字典、权限数据等（需要缓存/分页/刷新）。
- **Zustand**：主题、布局、多标签页、临时交互状态、登录态（token）等。

（详见《前端状态管理设计》。）

---

## 3. 架构约束（推荐）

1. **路由与权限**：动态路由以“后端菜单树/路由配置”为唯一来源；路由守卫只做登录态与权限校验。
2. **接口层隔离**：页面禁止直接 new Axios；必须通过统一 client 与 API 模块调用。
3. **统一响应体适配**：前端需对接后端 `R<T>`（或 `Result<T>`）结构，集中做解包与错误码映射。
4. **可观测性（前端）**：所有请求带 `X-Request-Id`（可在 Axios 拦截器生成/注入），便于前后端链路排障。
